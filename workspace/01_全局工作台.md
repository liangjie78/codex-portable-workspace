本文档是 `{{WORKSPACE_ROOT}}` 的唯一项目工作流入口，承接 Codex 全局规范中的跳转入口，并集中维护项目工作台、项目创建 SOP 和 CodexMemory 入口规则。

正式存放位置：

```text

## 最高优先级工作规则

1. `{{WORKSPACE_ROOT}}` 是唯一正式工作区。
2. 当前 C 盘 Codex 会话目录只用于运行会话和临时操作，不作为正式项目存放位置。
3. 新项目必须创建独立项目文件夹。
4. 项目文件夹命名必须使用 `Project-编号-项目名称`。
5. 项目相关的需求、草稿、调研、源码、素材、输出、日志、复盘全部放入对应项目文件夹。
6. 不得把项目相关内容散落到桌面、下载目录、用户主目录、C 盘 Documents/Codex 默认目录、临时目录或其他非项目目录。
7. 如果任务不是新项目，而是维护、排查、修改已有项目，必须先定位 `{{WORKSPACE_ROOT}}\Projects\` 下对应项目文件夹，再在该文件夹内继续工作。
8. 如果无法判断当前任务属于哪个项目，先暂停并向用户确认，不要自行把文件放到 Workspace 根目录或 C 盘默认目录。
9. 任务涉及本机工具、运行环境、依赖、服务、端口、数据库、缓存、浏览器验证、构建或 Windows/PowerShell 细节时，开始前应按需阅读 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`。
10. 任务过程中一旦发现可复用的本机环境与工具变化，应及时更新 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`；项目完成时再做兜底复查。

## 协作执行三层策略

Codex 处理任务时，默认先按三层策略判断执行模式；只有建议启用 CC-Switch worker 或多 Agent 协作时，才需要把建议提交给用户审核：

1. Codex 直接处理：普通聊天、概念解释、简短问答、方案讨论、简单命令、低风险小改、单文件修改、阻塞性阅读或定位。默认不启用多 Agent 或 CC-Switch worker。
2. Codex + CC-Switch worker：边界清楚、风险可控、可验证交付物的默认省额度执行模式。Codex 负责监督、决策、审查和最终验收；worker 只负责受限执行。
3. 多 Agent 协作：显式高级模式，不因“中大型项目”或“任务复杂”自动启用。只有用户明确要求，或确实需要 3 个及以上互补专家岗位并行产出、方案竞争或交叉复核时，才建议启用。

复杂任务默认先由 Codex 拆解、控范围、定验收。拆解后如果出现清晰执行块，优先考虑 Codex + CC-Switch worker；只有需要多个独立专家视角同时成立时，才考虑多 Agent。

如果判断为 Codex 直接处理，不必为了模式选择打断用户；如果建议启用 worker 或多 Agent，应先说明建议模式、理由、关键风险和将读取的规则文件，由用户确认后再启用。

```

启用多 Agent 协作时，读取并遵守：

## CodexMemory RAG 入口

`{{WORKSPACE_ROOT}}\CodexMemory` 是本机 Codex 长期记忆知识库，保存可按需检索的本机环境、工具经验、Windows/PowerShell 踩坑、项目历史、构建测试流程和常见报错。

使用边界：

- 全局硬规则、安全边界、项目目录规则和任务启动规则仍保留在全局规范与本文档中，不依赖 RAG 才能生效。
- 任务涉及已有项目、本机环境、Windows/PowerShell、历史报错、构建测试部署、MCP/CLI/浏览器验证或可复用工具经验时，可优先查询 CodexMemory。
- 检索结果只作为参考；涉及命令、路径、端口、版本、服务状态或外部系统时必须重新验证。
- 需要排查 CodexMemory 自身、批量沉淀经验或索引异常时，先调用只读 `rag_health`。若索引不是 `current`、有校验错误，或报告绝对本机来源缺失，先修复证据/执行 `rag_reindex`，再继续写入知识卡。证据已实际复核后，调用 `rag_mark_verified` 刷新日期；搜索或读取卡片不等于复核。
- 对大规模卡片迁移、结构变更或人工批量编辑，先运行 `rag_snapshot` 创建带 SHA-256 清单的本地恢复点；该工具只创建快照，不会自动恢复、覆盖或删除知识数据。
- 不得把密钥、token、cookie、账号密码、生产地址、项目私有连接串或一次性临时命令写入 CodexMemory。

相关 MCP/CLI 源码位于：

```text
{{WORKSPACE_ROOT}}\Projects\Project-013-CodexMemory\03_Source\codex-memory-mcp
```

本机知识库数据和 MCP 源码必须保持分离。MCP 是可迁移工具层；知识库内容只与当前电脑关联，换电脑时应重新初始化或由用户明确执行备份/导入。

## 什么时候启动项目工作流

满足以下任一条件时，默认视为新项目，应启动项目工作流并执行项目文件夹规则：

- 需要创建或长期保存多个文件、源码、素材、文档、输出或日志。
- 需要多轮规划、实现、验证、交付或复盘。
- 需要建立独立项目目录、运行环境、仓库、页面、服务、脚本、文档集或交付包。
- 用户明确说“新项目”“项目”“做一个项目”“创建一个”“搭建一个系统/网站/工具/平台/应用”“设计一套”“整理一套”等。
- 任务会产生可复用资产，未来可能继续迭代、维护或扩展。

通常不视为新项目的情况：

- 简单问答、一次性命令、临时排查、单文件小改、纯解释说明。
- 不需要保存文件的临时建议。
- 用户明确要求不要创建文件。

如果任务边界很小、无长期资产、用户没有要求纳入项目管理，Codex 可以不启动项目工作流。

## 前置文档缺失时

如果本文档不存在、不可读或内容明显损坏，Codex 不得静默跳过。应先告知用户缺失项和影响；若任务仍需继续，应在用户确认后按最小兜底流程推进，并把缺失情况记录到项目日志或最终回复。

## 项目编号规则

项目统一放在：

```text
{{WORKSPACE_ROOT}}\Projects\
```

命名格式：

```text
Project-三位编号-项目名称
```

编号生成步骤：

1. 查看 `{{WORKSPACE_ROOT}}\Projects\` 下已有项目。
2. 找到最大编号。
3. 新项目编号 = 最大编号 + 1。
4. 如果没有任何项目，从 `001` 开始。
5. 编号固定三位，不足补零。
6. 不复用已经使用过的编号。

项目名称规则：

- 使用简短、可识别名称。
- 可以使用中文，但涉及 MCP、CLI、构建工具或跨进程传参时，优先使用 ASCII 项目名。
- 不使用 Windows 非法字符：`\ / : * ? " < > |`。
- 避免含糊名称，如 `测试`、`新项目`、`资料`。

## 标准创建流程

### Step 1 - 启动前确认

开始新项目之前必须阅读本文档：

```text
{{WORKSPACE_ROOT}}\01_全局工作台.md
```

如果项目涉及本机工具、运行环境、依赖、服务、端口、数据库、缓存、浏览器验证、构建、部署或 Windows/PowerShell 细节，还应按需阅读：

```text
{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md
```

### Step 2 - 创建项目文件夹

在 `{{WORKSPACE_ROOT}}\Projects\` 下创建：

```text
Project-编号-项目名称\
```

### Step 3 - 创建标准目录

每个项目默认创建：

```text
Project-编号-项目名称\
├── 00_Project_Brief.md
├── 01_Requirements\
├── 02_Research\
├── 03_Source\
├── 04_Assets\
├── 05_Outputs\
├── 06_Logs\
└── 99_Retrospective.md
```

目录含义：

- `00_Project_Brief.md`：项目目标、边界、验收标准、关键约束。
- `01_Requirements\`：需求、用户输入、范围变更。
- `02_Research\`：调研、资料、竞品、方案比较。
- `03_Source\`：源码、脚本、配置、可编辑源文件。
- `04_Assets\`：图片、字体、音频、视频、素材。
- `05_Outputs\`：交付物、导出文件、最终结果。
- `06_Logs\`：执行日志、问题记录、过程笔记。
- `99_Retrospective.md`：项目复盘、经验沉淀、可复用方法。

### Step 4 - 写入项目简报

`00_Project_Brief.md` 必须包含：

```markdown
# Project-编号-项目名称

## 项目目标

## 用户需求

## 范围边界

## 关键约束

## 交付物

## 验收标准

## 当前状态
```

### Step 5 - 存放规则

所有项目相关内容必须进入项目文件夹。

禁止：

- 放到桌面。
- 放到下载目录。
- 放到用户主目录。
- 放到 C 盘 Documents/Codex 默认目录。
- 放到当前 Codex 会话目录。
- 长期放在临时目录。
- 放到不相关项目文件夹。

### Step 6 - 项目执行中维护日志

执行过程中，如果发生以下情况，写入 `06_Logs\`：

- 重要决策。
- 用户变更需求。
- 安装依赖。
- 遇到错误。
- 路径、权限、环境问题。
- 测试结果。
- 未解决问题。

### Step 7 - 项目结束复盘

项目结束时更新：

```text
99_Retrospective.md
```

至少包含：

- 做成了什么。
- 没做什么。
- 遇到哪些问题。
- 哪些方法以后可复用。
- 哪些经验要沉淀到 `{{WORKSPACE_ROOT}}\CodexMemory`。

如果某条经验对未来项目通用，写入 `{{WORKSPACE_ROOT}}\CodexMemory` 的知识卡片，不再追加到本文档。

### Step 8 - 及时维护本机环境与工具清单

项目执行过程中，一旦验证出新的本机环境信息，应及时更新工具清单，不要等项目结束才被动补录。

需要关注：

- 长期可用工具、路径、版本或运行时。
- 服务启动、停止、测试、构建、浏览器验证、数据库或缓存流程。
- Windows、PowerShell、编码、权限、端口、后台进程等注意事项。

如果具备跨项目复用价值，更新：

```text
{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md
```

不要写入项目私有配置、密钥、token、cookie、账号密码、生产地址或一次性临时命令。

项目结束时再做兜底复查。如果没有新增内容，应在项目复盘或最终回复中说明。

### 任务完成时必须做的判断

每次任务完成前，Codex 都要判断本次任务是否产生了可复用经验。重点看：

- 新发现的本机工具、路径、版本、启动方式、停止方式或环境限制。
- Windows、PowerShell、编码、端口、浏览器、MCP、CLI、worker 等踩坑。
- 项目执行中验证过的构建、测试、部署、排查或恢复流程。
- 可复用的需求判断、交付边界、验收方法或失败降级方案。

如果有可复用经验：

1. 优先通过 CodexMemory MCP `rag_finish_task` 写入 1 到 5 张知识卡。
2. 如 MCP 尚未可用，使用 CLI：`node src\cli.mjs finish <task-completion.json>` 或直接写卡后运行 `validate`、`reindex`。
3. 经验卡必须包含来源、适用范围、风险和证据。
4. 涉及本机工具和环境变化时，同时按需更新 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`。
5. 写入后调用 `rag_health`；只有校验通过且索引为 `current` 时，才把本次沉淀视为完成。

如果没有可复用经验：

- 不写占位卡。
- 在项目复盘或最终回复中说明本次未发现需要沉淀到 CodexMemory 的新经验。
