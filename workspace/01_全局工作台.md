# 全局工作台与项目工作流

本文件是 `{{WORKSPACE_ROOT}}` 唯一的项目工作流入口，负责新项目判定、目录、创建、记录和收尾。通用工程、安全、沟通与执行模式规则由当前智能体的用户级入口提供：Codex 使用 `{{CODEX_AGENTS}}`，Claude Code 使用 `%USERPROFILE%\.claude\CLAUDE.md`。

## 工作区硬规则

1. `{{WORKSPACE_ROOT}}` 是唯一正式工作区；C 盘 Codex 会话目录只用于运行会话和临时操作。
2. 新项目必须使用独立目录：`{{WORKSPACE_ROOT}}\Projects\Project-三位编号-项目名称\`。
3. 需求、调研、源码、素材、输出、日志和复盘全部放入对应项目目录，不散落到桌面、下载、用户主目录、会话目录或 Workspace 根目录。
4. 维护已有项目时，先定位 `{{WORKSPACE_ROOT}}\Projects\` 下的项目目录。
5. 无法判断任务归属时先问用户，不自行创建或移动项目内容。
6. 涉及本机环境与工具时，先按需阅读 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`。
7. 本文件缺失、不可读或损坏时必须告知用户；经确认后才能按最小兜底流程继续。

## 何时创建项目

满足任一条件时，默认启动项目工作流：

- 需要长期保存多个文件、源码、素材、文档、输出或日志。
- 需要多轮规划、实现、验证、交付或复盘。
- 需要独立目录、运行环境、仓库、页面、服务、脚本、文档集或交付包。
- 用户明确要求新建项目、系统、网站、工具、平台、应用或一套可复用资产。

简单问答、一次性命令、临时排查、单文件小改、纯说明，或用户明确要求不创建文件时，通常不启动项目工作流。

## 启动前检查

新项目开始前：

1. 阅读本文件。
2. 如涉及本机工具、运行环境、依赖、服务、端口、数据库、缓存、浏览器、构建、部署或 PowerShell，阅读 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`。
3. 检查项目根目录的 `README.md`、`AGENTS.md`、`00_Project_Brief.md` 和 `99_Retrospective.md`；小写文件视为等价。
4. 新项目按本文模板创建四份核心文档，填写可推断的项目名、日期、技术栈、包管理器和主要目录；无法推断的必填项一次性询问用户。
5. 在项目规范和项目简报就绪前，不修改项目代码或外部状态。

## 四份核心文档

四份文件保留，职责不合并：

| 文件 | 读者 | 唯一职责 |
|---|---|---|
| `README.md` | 使用者、开发者、访客 | 项目介绍、使用方式和文档入口 |
| `AGENTS.md` | Codex 与其他代码智能体 | 项目命令、约定、修改边界和验证规则 |
| `00_Project_Brief.md` | 项目负责人、执行者 | 启动基线：目标、范围、约束和验收 |
| `99_Retrospective.md` | 维护者、未来项目 | 实际结果、偏差、问题和经验 |

去重规则：

- 项目介绍只写在 `README.md`。
- 目标、范围和验收标准只写在 `00_Project_Brief.md`。
- 执行规则只写在 `AGENTS.md`。
- 实际结果和经验只写在 `99_Retrospective.md`。
- 其他文件需要相关内容时只写一句摘要并链接原文。
- `AGENTS.md` 保存完整开发命令；`README.md` 只保留使用者需要的安装、运行和测试命令。
- 复盘中仍然有效的执行规则提炼到 `AGENTS.md`；面向使用者的变化更新到 `README.md`，不复制整段复盘。

## 编号与命名

项目目录格式：

```text
Project-三位编号-项目名称
```

编号规则：检查 `{{WORKSPACE_ROOT}}\Projects\` 的历史最大编号，新编号加 1；没有项目时从 `001` 开始。编号固定三位且不复用。

项目名应简短、可识别。可用中文；涉及 MCP、CLI、构建或跨进程传参时优先 ASCII。禁止 Windows 非法字符 `\ / : * ? " < > |`，避免“测试”“新项目”“资料”等含糊名称。

## 创建项目 SOP

1. 确认项目编号、名称和目录。
2. 创建标准子目录。
3. 创建 `README.md`、`AGENTS.md`、`00_Project_Brief.md` 和 `99_Retrospective.md`。
4. 按下方模板填写四份核心文档；删除不适用的占位提示，不保留空泛内容。
5. 检查四份文件是否职责清楚、链接有效且没有大段重复。
6. 在 `00_Project_Brief.md` 的目标、范围、交付物和验收标准明确后开始实现。

PowerShell 参考：

```powershell
$project = "Project-001-项目名称"
$root = "{{WORKSPACE_ROOT}}\Projects\$project"
New-Item -ItemType Directory -Force -Path `
  "$root\01_Requirements", "$root\02_Research", "$root\03_Source", `
  "$root\04_Assets", "$root\05_Outputs", "$root\06_Logs" | Out-Null
New-Item -ItemType File -Force -Path `
  "$root\README.md", "$root\AGENTS.md", "$root\00_Project_Brief.md", `
  "$root\99_Retrospective.md" | Out-Null
```

## 执行记录

以下内容写入 `06_Logs\`：

- 重要决策和需求变更。
- 依赖安装、错误、路径、权限和环境问题。
- 测试结果与未解决事项。

知识库使用 GBrain MCP 的 `obsidian` source。写入须遵守 `20_Wiki/知识库治理规范.md`：单份资料进入 `10_Notes/`，跨资料结论才进入 `20_Wiki/`；只有用户确认进入 `00_Inbox/Trusted/` 的资料可自动整理。Obsidian 轻量助手转交的任务从 `00_Inbox/Codex-Handoff/` 接管，Codex 须重新验证后再执行。

检索与沉淀细则以 Vault 的 `20_Wiki/Codex-Obsidian-GBrain-知识工作流.md` 为准。日常任务默认只走一次精确读取或关键词检索，写入经过复用门槛；Codex 创建或更新知识页时优先使用 GBrain `put_page`，写后回读，不直接制造未索引的 Markdown。

## 项目收尾

项目完成后更新 `99_Retrospective.md`，至少记录：

- 已完成和未完成内容。
- 遇到的问题与剩余风险。
- 可复用方法和后续建议。
- 知识库和工具清单的更新情况。

收尾时同步检查：

- `README.md` 是否反映当前功能、运行方式和限制。
- `AGENTS.md` 是否保留仍有效的项目规则，并删除过期命令。
- `00_Project_Brief.md` 是否只记录启动基线和已确认的范围变更。
- 四份核心文档是否仍符合去重规则。

通用经验按知识库治理规范进入来源笔记或 Wiki；不要继续堆入本工作流文件。

项目连续 3 天无文件改动时，视为阶段性静默。Codex 只在任务结束时检查该状态，自动把目标、结果、关键决策、踩坑、验证和未完成项整理为知识库项目历史页；项目再次活跃后复用并更新同一页。该检查不要求后台定时服务。

## GBrain 运行约定

Codex 通过 `http://127.0.0.1:3131/mcp` 使用共享 GBrain 服务，source 为 `obsidian`。监督器按请求同步 Vault 并启动唯一 PGlite 后端；后端空闲 30 分钟后退出，下次请求自动恢复。

- 修改 MCP 配置或重新登录后重启 Codex，并实际读取 `20_wiki/知识库治理规范` 验证 source。
- 监督器只在后端启动前同步 Vault；后端在线时不会持续监听文件。Obsidian Git 提交只负责版本记录，不会触发在线后端重同步。人工新建的笔记在后端空闲退出、下一次请求冷启动时进入索引；需要立即检索时由 Codex 通过 `put_page` 写入。
- 后端运行时不要另起 `gbrain serve`、`sync` 或 `dream`，不要删除锁文件。
- `GBrain Dream` 每天 02:00 执行；后端在线时会跳过。手动运行前先等待后端释放。
