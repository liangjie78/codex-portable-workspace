# 全局工作台与项目工作流

本文档是 `{{WORKSPACE_ROOT}}` 的唯一项目工作流入口，承接 Codex 全局规范中的跳转入口，并集中维护项目工作台、项目创建 SOP、全局复利与踩坑日志。

正式存放位置：

```text
{{WORKSPACE_ROOT}}\01_全局工作台.md
```

## 最高优先级工作规则

1. `{{WORKSPACE_ROOT}}` 是唯一正式工作区。
2. 当前 C 盘 Codex 会话目录只用于运行会话和临时操作，不作为正式项目存放位置。
3. 新项目必须创建独立项目文件夹。
4. 项目文件夹命名必须使用 `Project-编号-项目名称`。
5. 项目相关的需求、草稿、调研、源码、素材、输出、日志、复盘全部放入对应项目文件夹。
6. 不得把项目相关内容散落到桌面、下载目录、用户主目录、C 盘 Documents/Codex 默认目录、临时目录或其他非项目目录。
7. 如果任务不是新项目，而是维护、排查、修改已有项目，必须先定位 `{{WORKSPACE_ROOT}}\Projects\` 下对应项目文件夹，再在该文件夹内继续工作。
8. 如果无法判断当前任务属于哪个项目，先暂停并向用户确认，不要自行把文件放到 Workspace 根目录或 C 盘默认目录。
9. 任务涉及本机工具、运行环境、依赖、服务、端口、数据库、缓存、浏览器验证、构建或 Windows/PowerShell 细节时，开始前应按需阅读 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`。
10. 任务过程中一旦发现可复用的本机环境与工具变化，应及时更新 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`；项目完成时再做兜底复查。

## 协作执行三层策略

Codex 处理任务时，默认先按三层策略判断执行模式；只有建议启用 CC-Switch worker 或多 Agent 协作时，才需要把建议提交给用户审核：

1. Codex 直接处理：普通聊天、概念解释、简短问答、方案讨论、简单命令、低风险小改、单文件修改、阻塞性阅读或定位。默认不启用多 Agent 或 CC-Switch worker。
2. Codex + CC-Switch worker：边界清楚、风险可控、可验证交付物的默认省额度执行模式。Codex 负责监督、决策、审查和最终验收；worker 只负责受限执行。
3. 多 Agent 协作：显式高级模式，不因“中大型项目”或“任务复杂”自动启用。只有用户明确要求，或确实需要 3 个及以上互补专家岗位并行产出、方案竞争或交叉复核时，才建议启用。

复杂任务默认先由 Codex 拆解、控范围、定验收。拆解后如果出现清晰执行块，优先考虑 Codex + CC-Switch worker；只有需要多个独立专家视角同时成立时，才考虑多 Agent。

如果判断为 Codex 直接处理，不必为了模式选择打断用户；如果建议启用 worker 或多 Agent，应先说明建议模式、理由、关键风险和将读取的规则文件，由用户确认后再启用。

启用 CC-Switch worker 时，读取并遵守：

```text

```

启用多 Agent 协作时，读取并遵守：

```text

```

## 什么时候启动项目工作流

满足以下任一条件时，默认视为新项目，应启动项目工作流并执行项目文件夹规则：

- 需要创建或长期保存多个文件、源码、素材、文档、输出或日志。
- 需要多轮规划、实现、验证、交付或复盘。
- 需要建立独立项目目录、运行环境、仓库、页面、服务、脚本、文档集或交付包。
- 用户明确说“新项目”“项目”“做一个项目”“创建一个”“搭建一个系统/网站/工具/平台/应用”“设计一套”“整理一套”等。
- 任务会产生可复用资产，未来可能继续迭代、维护或扩展。

通常不视为新项目的情况：

- 简单问答、一次性命令、临时排查、单文件小改、纯解释说明。
- 不需要保存文件的临时建议。
- 用户明确要求不要创建文件。

如果任务边界很小、无长期资产、用户没有要求纳入项目管理，Codex 可以不启动项目工作流。

## 前置文档缺失时

如果本文档不存在、不可读或内容明显损坏，Codex 不得静默跳过。应先告知用户缺失项和影响；若任务仍需继续，应在用户确认后按最小兜底流程推进，并把缺失情况记录到项目日志或最终回复。

## 项目编号规则

项目统一放在：

```text
{{WORKSPACE_ROOT}}\Projects\
```

命名格式：

```text
Project-三位编号-项目名称
```

编号生成步骤：

1. 查看 `{{WORKSPACE_ROOT}}\Projects\` 下已有项目。
2. 找到最大编号。
3. 新项目编号 = 最大编号 + 1。
4. 如果没有任何项目，从 `001` 开始。
5. 编号固定三位，不足补零。
6. 不复用已经使用过的编号。

项目名称规则：

- 使用简短、可识别名称。
- 可以使用中文，但涉及 MCP、CLI、构建工具或跨进程传参时，优先使用 ASCII 项目名。
- 不使用 Windows 非法字符：`\ / : * ? " < > |`。
- 避免含糊名称，如 `测试`、`新项目`、`资料`。

## 标准创建流程

### Step 1 - 启动前确认

开始新项目之前必须阅读本文档：

```text
{{WORKSPACE_ROOT}}\01_全局工作台.md
```

如果项目涉及本机工具、运行环境、依赖、服务、端口、数据库、缓存、浏览器验证、构建、部署或 Windows/PowerShell 细节，还应按需阅读：

```text
{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md
```

### Step 2 - 创建项目文件夹

在 `{{WORKSPACE_ROOT}}\Projects\` 下创建：

```text
Project-编号-项目名称\
```

### Step 3 - 创建标准目录

每个项目默认创建：

```text
Project-编号-项目名称\
├── 00_Project_Brief.md
├── 01_Requirements\
├── 02_Research\
├── 03_Source\
├── 04_Assets\
├── 05_Outputs\
├── 06_Logs\
└── 99_Retrospective.md
```

目录含义：

- `00_Project_Brief.md`：项目目标、边界、验收标准、关键约束。
- `01_Requirements\`：需求、用户输入、范围变更。
- `02_Research\`：调研、资料、竞品、方案比较。
- `03_Source\`：源码、脚本、配置、可编辑源文件。
- `04_Assets\`：图片、字体、音频、视频、素材。
- `05_Outputs\`：交付物、导出文件、最终结果。
- `06_Logs\`：执行日志、问题记录、过程笔记。
- `99_Retrospective.md`：项目复盘、经验沉淀、可复用方法。

### Step 4 - 写入项目简报

`00_Project_Brief.md` 必须包含：

```markdown
# Project-编号-项目名称

## 项目目标

## 用户需求

## 范围边界

## 关键约束

## 交付物

## 验收标准

## 当前状态
```

### Step 5 - 存放规则

所有项目相关内容必须进入项目文件夹。

禁止：

- 放到桌面。
- 放到下载目录。
- 放到用户主目录。
- 放到 C 盘 Documents/Codex 默认目录。
- 放到当前 Codex 会话目录。
- 长期放在临时目录。
- 放到不相关项目文件夹。

### Step 6 - 项目执行中维护日志

执行过程中，如果发生以下情况，写入 `06_Logs\`：

- 重要决策。
- 用户变更需求。
- 安装依赖。
- 遇到错误。
- 路径、权限、环境问题。
- 测试结果。
- 未解决问题。

### Step 7 - 项目结束复盘

项目结束时更新：

```text
99_Retrospective.md
```

至少包含：

- 做成了什么。
- 没做什么。
- 遇到哪些问题。
- 哪些方法以后可复用。
- 哪些坑要写入本文档的“全局复利与踩坑日志”。

如果某条经验对未来项目通用，追加到本文档的“全局复利与踩坑日志”。

### Step 8 - 及时维护本机环境与工具清单

项目执行过程中，一旦验证出新的本机环境信息，应及时更新工具清单，不要等项目结束才被动补录。

需要关注：

- 长期可用工具、路径、版本或运行时。
- 服务启动、停止、测试、构建、浏览器验证、数据库或缓存流程。
- Windows、PowerShell、编码、权限、端口、后台进程等注意事项。

如果具备跨项目复用价值，更新：

```text
{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md
```

不要写入项目私有配置、密钥、token、cookie、账号密码、生产地址或一次性临时命令。

项目结束时再做兜底复查。如果没有新增内容，应在项目复盘或最终回复中说明。

## 全局复利与踩坑日志

本节用于记录犯过的错误、踩过的坑、走过的弯路，以及未来可以持续复用的方法。目标是让每次失败都能沉淀成下次更快、更稳的路径。

### 使用规则

1. 每次遇到明显错误、返工、误判、环境问题、工具问题，都要记录。
2. 每次完成项目复盘时，提取通用经验写入本节。
3. 记录要具体，不写空泛总结。
4. 优先记录可操作的“以后怎么做”。
5. 如果经验只适用于某个项目，先写入对应项目的 `99_Retrospective.md`；如果可复用，再同步到本节。
6. 任务涉及本机环境与工具时，开始前应按需阅读 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`；任务过程中发现新的可复用工具、路径、版本、启动方式、停止方式、端口、Windows 注意事项或环境限制时，应及时更新工具清单。

### 记录格式

每条记录使用下面模板：

```markdown
## YYYY-MM-DD - 简短标题

- 项目/上下文：
- 发生了什么：
- 根因：
- 影响：
- 当时走过的弯路：
- 正确做法：
- 未来复用规则：
- 相关文件/证据：
```

### 坑位索引

- 环境与安装：Python、Node、MCP、依赖、权限、路径、系统差异导致的问题。
- 文件与目录：文件散落、目录混乱、命名不一致、交付物放错位置等问题。
- 需求理解：误解用户目标、范围不清、没有先问关键问题导致返工的问题。
- 工程实现：代码结构、测试、构建、部署、兼容性、性能相关问题。
- 设计与内容：UI/UX、PPT、文档、视觉、文案、信息架构相关问题。
- 工具使用：Codex、MCP、插件、skill、CLI、浏览器、GitHub、Office 工具等使用经验。
- 本机环境与工具清单：具备跨项目复用价值的本机环境经验应同步到 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`。

### 可复用方法库

#### 新项目先建目录

规则：任何新项目开始前，先创建 `{{WORKSPACE_ROOT}}\Projects\Project-编号-项目名称\`，再开始写文件。

原因：项目资产一旦散落，后续查找、复盘、交付都会变慢。

#### D 盘 Workspace 是正式位置

规则：所有正式项目都放在 `{{WORKSPACE_ROOT}}`。C 盘 Codex 默认目录只作为会话运行目录或临时环境，不作为正式项目资产存放位置。

#### 临时文件和长期文件分离

规则：

- 临时实验放 `{{WORKSPACE_ROOT}}\Temp\`。
- 项目最终交付放项目目录下的 `05_Outputs\`。
- 跨项目或集中查看产物可放 `{{WORKSPACE_ROOT}}\Global_Outputs\`。
- 长期源文件、素材、日志必须在项目目录内。

#### 任务过程中及时维护本机工具清单

规则：涉及本机环境与工具的任务，开始前按需阅读 `{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md`；任务过程中一旦发现可跨项目复用的本机环境信息，应及时写入工具清单。

不写入：

- 项目私有配置。
- 密钥、token、cookie、账号密码、生产地址。
- 一次性临时命令。

项目完成时再做兜底复查。如果没有新增内容，应在项目复盘或最终回复中说明。

## 2026-06-28 - 安装 CLI 工具时先确认配置写入作用域

- 项目/上下文：安装 Agent Reach 及其上游工具 `mcporter`。
- 发生了什么：在 Codex 会话工作区运行 `mcporter config add exa ...` 时，`mcporter` 默认把配置写入当前目录的 `config\mcporter.json`，与 Agent Reach 安装文档“不要污染工作区”的目录规则冲突。
- 当时走过的弯路：先按文档的简写命令执行，才发现该 CLI 默认作用域是 project；随后确认 `mcporter config add <name> <url> --scope home` 才会写入用户级 `C:\Users\Administrator\.mcporter\mcporter.json`。
- 正确做法：安装会写配置的全局 CLI/MCP 工具时，优先从用户目录或工具专用目录执行，并先查看是否支持 `--scope home`、`--persist`、`--config` 这类参数；如果误写工作区配置，应确认路径后只清理本次误生成的文件。
- 未来复用规则：不要假设“全局安装的 CLI”就一定写全局配置；执行任何 `config add` 类命令前先确认作用域和落盘路径。
- 相关文件/证据：`{{WORKSPACE_ROOT}}\00_本机环境与工具清单.md` 的 Agent Reach 记录。

## 2026-06-21 - 配置仓库必须用远端克隆证明可恢复

- 项目/上下文：Project-008-Codex-Portable-Workspace，将 Codex 和 Workspace 规则迁移到私有 GitHub 仓库。
- 发生了什么：本地脚本和隔离安装通过后，仍需要证明 GitHub 上的实际提交没有漏文件、编码变化或推送差异。
- 根因：验证原工作树只能证明本地文件可用，不能证明远端仓库本身是有效备份。
- 影响：如果不做远端克隆演练，可能在换电脑时才发现未跟踪文件、`.gitignore` 误排除、编码损坏或推送未完成。
- 当时走过的弯路：隔离 worker doctor 把当前真实会话中的旧路径存活进程识别为 drift；这不是恢复失败，而是进程态检查不能在两套安装同时运行时混用。
- 正确做法：本地运行 secret scan 和完整验证；从私有远端重新 clone；从 clone 安装到第二个隔离目录；用 Codex 实际解析生成配置；最后比较本地与远端 SHA 和仓库可见性。
- 未来复用规则：任何“备份/可迁移配置仓库”完成前，都必须进行一次从远端重新克隆的灾难恢复演练。静态文件验证、安装验证和真实运行时验证应分层进行。
- 相关文件/证据：
  - `{{WORKSPACE_ROOT}}\Projects\Project-008-Codex-Portable-Workspace\06_Logs\verification.md`
  - `https://github.com/liangjie78/codex-portable-workspace`

## 2026-06-21 - Windows 安装 GitHub skill 后必须验证符号链接目标

- 项目/上下文：把 `ui-ux-pro-max` 从 Codex 同步安装到 Claude Code。
- 发生了什么：通用 GitHub skill 安装器显示安装成功，但仓库里的 `data`、`scripts` 是符号链接，Windows 下载后变成了只包含相对路径的普通文本文件。
- 根因：GitHub 内容下载和 Windows 文件系统没有还原仓库符号链接语义。
- 影响：`SKILL.md` 存在且目录看似完整，但实际 Python 搜索脚本和数据集不可用。
- 当时走过的弯路：只根据安装器的 “Installed” 输出判断成功。
