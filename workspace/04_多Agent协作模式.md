# 多 Agent 协作模式

启用多 Agent 后必须遵守本文件。启动条件见 `{{CODEX_AGENTS}}`。

## 定位与启用

多 Agent 是显式高级模式，用于认知分工、方案竞争和独立复核，不是普通工程执行或省额度工具。复杂项目不会自动启用。

除非用户已明确要求多 Agent 协作，否则 Codex 必须先说明：

- 启用理由、岗位及各自交付物。
- 单 Agent 或 CC-Switch worker 为什么不够。
- 线程与额度成本、关键风险和降级方案。
- 将读取并遵守本文件。

用户确认后才能编组。任务必须能拆出至少 3 个职责清楚、互补且有独立价值的子 Agent；否则由 Codex 直接处理。CC-Switch worker 是受限执行工具，不计入子 Agent 数量。

## 主控关系

- Codex 是唯一 PM、决策者和最终验收者。
- 子 Agent 提供专家判断、阶段成果或独立审查。
- CC-Switch worker 只执行边界明确、可回滚、可验证的小块工作。
- 任何 Agent 或 worker 的输出都不能绕过 Codex 成为最终交付。
- 高风险判断、冲突处理、结果整合和最终验证由 Codex 负责。

项目创建、目录、日志与复盘遵守 `{{WORKSPACE_ROOT}}\01_全局工作台.md`；worker 调用遵守 `{{WORKSPACE_ROOT}}\05_CC-Switch-Worker-MCP规则.md`。

## 启动准备

用户提供需求清单、功能列表或零散目标时，先整理成项目 Brief，再编组。至少包含：

- 目标用户、场景和项目目标。
- 功能范围、非功能范围和 MVP 边界。
- 约束、风险、假设和验收标准。
- 阶段、交付物、岗位及协作关系。

不影响启动的缺口可明确假设；会改变目标或范围的缺口先提问。

编组说明应简短列出：启用原因、岗位、职责边界、交付物、依赖关系、失败时的 PM 接管方式，以及如何避免重复工作。

## 岗位规则

- 岗位按真实职责命名，如产品经理、架构师、前端工程师、测试工程师、安全审查员、研究员或文档工程师；不得使用“Agent 1”等无意义名称。
- 工具自动分配随机昵称时，立即建立“系统昵称 -> 岗位名称”映射；用户可见内容使用“岗位名称（系统昵称）”。
- 每个岗位只有一个主责，职责、输入、输出和禁区必须明确。
- 同一问题只设一个主责岗位，其他岗位最多补充审查。
- 岗位随阶段增减；完成后及时释放，避免空转和占用线程。
- 子 Agent 不得扩大范围、修改未授权文件或直接向用户交付最终结论。

常用岗位按需选择：

- 产品与研究：产品经理、研究员、数据分析师、商业或合规审查员。
- 技术与实现：技术负责人、架构师、前端、后端、数据库、DevOps、性能工程师。
- 质量与交付：测试工程师、安全审查员、Review 工程师、文档工程师、质量审校员。
- 设计与内容：UI/UX、视觉、品牌、内容策划、原型设计师、创意总监。

不要为了凑岗位而启用无增量价值的角色。

## 派发标准

每个子任务必须具体、有限、可验收，任务说明包含：

- 项目路径、背景和目标。
- 允许与禁止修改的文件或目录。
- 明确禁区。
- 验收标准和验证命令。
- 必须回传的内容。

“优化整个项目”“你看着改”“全面重构”“修复所有问题”等任务必须先拆解。

避免重复消耗：不同 Agent 不做相同调研，不重复读取同一大文件；派发前确认输出用途；回传只保留结论、证据、改动、验证和风险。

## 并行修改

- 默认禁止多个 Agent 同时修改同一文件或同一批核心文件。
- 并行任务必须有互不重叠的文件边界。
- Codex 统一 review、整合和验证；冲突时不得自动覆盖任一方。
- 发现用户、外部工具或其他 Agent 的并发改动时，先区分来源。无冲突则基于最新文件继续；有冲突则阅读并适配，无法安全合并时告知用户。
- 未经明确要求，不覆盖、回滚或删除外部改动。最终回复说明观察到的并发修改及处理方式。

## 回传与验收

子 Agent 回传：

- 完成内容、修改文件和原因。
- 是否越界。
- 已运行与未运行的验证。
- 剩余风险、疑问和需要 Codex 判断的事项。

Codex 必须检查：

- 文件范围和用户目标是否一致。
- 是否有无关重构、依赖或配置变化。
- 是否触及数据库、CI、权限、部署等高风险表面。
- 是否删除、弱化、跳过或伪造测试。
- 是否符合项目架构与风格，是否缺测试、文档或日志。

只有 Codex 审查并完成最终验证后，任务才算完成。

## 失败与降级

- 按阶段和依赖分批创建 Agent，不一次性无边界启动全部岗位。
- Agent 因线程、额度、工具或超时失败时，记录岗位缺口、影响和风险。
- 非核心岗位失败可继续推进，并在交付中说明缺口。
- 核心岗位失败由 Codex 接管；必要时把任务缩小后重新派发。
- 不让单个失败岗位阻塞整条主线。
- 失败、接管、跳过、重派和风险接受写入 `06_Logs/agent-review.md` 或对应项目日志。

## 协作日志

启用多 Agent 时维护：

```text
06_Logs/
  agent-handoffs.md   # 派发任务
  agent-decisions.md  # 方案与取舍
  agent-review.md     # 审查、失败与降级
```

通用经验按 `{{WORKSPACE_ROOT}}\01_全局工作台.md` 的知识库与复盘规则处理，不堆回本文件。

## 高风险禁区

未经用户明确授权，任何子 Agent 都不得：

- 删除文件、重置 Git、覆盖用户改动。
- 修改全局规范、账号配置或关键安全策略。
- 升级依赖、切换技术栈或操作生产环境。
- 使用真实凭据、密钥或生产数据。
- 发布、部署、推送镜像或创建正式 release。
- 修改用户或其他 Agent 正在处理的文件。

## 最终交付

最终回复由 Codex 给出，说明启用的岗位及分工、整合内容、实际验证、未运行的检查、剩余风险和降级情况。Codex 对最终结果负责。
