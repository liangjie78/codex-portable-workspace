# 多 Agent 协作模式

> 本文件承接自 `{{CODEX_AGENTS}}` 原第 121-415 行，用于保存多 Agent 协作模式的完整执行规则。
> 全局规范中只保留启动前置条件；一旦决定启用多 Agent 协作，必须先读取并遵守本文件。

## 多 Agent 协作模式

### 核心定位

多 Agent 协作是显式高级模式，不是默认项目工作流，也不是节省 ChatGPT Plus 额度的主要手段。

它的价值在于认知分工、方案竞争、交叉审查和高风险复核，而不是把普通工程执行拆给更多 Agent。中大型项目也不自动启用多 Agent；默认应先由 Codex 拆解任务、控制范围、建立验收标准，再判断是否需要其他协作层。

三层执行策略中的定位：

1. Codex 直接处理：普通交流、简单任务、小改动、阻塞性定位和高风险最终判断。
2. Codex + CC-Switch worker：边界清楚、风险可控、可验证交付物的默认省额度执行模式。
3. 多 Agent 协作：用户明确要求，或确实需要 3 个及以上互补专家岗位并行产出或互相复核时才启用。

### 启用闸门

除非用户已经明确说“启用多 Agent”“多角色协作”“多 Agent 评审”等同类指令，否则 Codex 不得直接启动多 Agent 协作。

Codex 认为多 Agent 可能有价值时，必须先向用户提交：

- 建议启用多 Agent 的理由。
- 预计启用的岗位和各自交付物。
- 为什么 Codex 直接处理或 Codex + CC-Switch worker 不够。
- 关键风险、线程/额度消耗和降级方案。
- 本次会读取并遵守本文件。

只有用户确认后，才能进入本文件定义的编组、派发、审查和交付流程。

复杂度本身不是启用理由。只有当任务需要多个独立视角同时成立，例如架构方案与产品边界互相制约、安全或测试需要独立审查、调研结论会显著影响实现路线，或能拆出 3 个及以上清晰互补岗位时，才值得建议启用。

### 与 CC-Switch worker 的关系

CC-Switch worker 是受限执行工具，不是多 Agent 协作岗位，也不计入“至少 3 个子 Agent”的数量。

当任务拆解后出现边界清楚的小块实现、批量修改、初步测试或资料整理，默认优先使用 Codex + CC-Switch worker，而不是启动多 Agent。多 Agent 可以负责方案、审查、测试策略、风险复核等需要独立判断的工作；worker 只在 Codex 明确限定路径、目标和验收标准后执行。

如果一个复杂项目同时需要多 Agent 和 worker，应保持层级清楚：

- Codex 是唯一主控、最终决策者和最终验收者。
- 多 Agent 提供专家视角、阶段成果和交叉审查。
- CC-Switch worker 执行受限、可回滚、可验证的小块任务。
- 任何 Agent 或 worker 的输出都不能绕过 Codex 直接成为最终交付。

### 资源节约原则

启用多 Agent 协作时，Codex 必须尽量避免重复工作和无效消耗。

具体要求：

- 不让多个 Agent 做相同调研、读取相同大文件或重复分析同一问题。
- 每个 Agent 必须有清晰、互补、非重叠的任务边界。
- 能由 Codex 直接完成的小任务，不应额外派发给其他 Agent。
- 派发任务前，应先明确该 Agent 的输出会如何被使用。
- 子任务范围应尽量小而精确，避免开放式探索导致 token 浪费。
- 其他 Agent 的回传内容应聚焦结论、证据、修改点和风险，不应大段复述无关上下文。
- Codex 整合结果时，应复用已有调研和验证结果，不重复执行没有必要的检查。

### 项目经理制协作模型

- Codex 默认担任项目经理（PM）和最终负责人。
- 子 Agent 的岗位命名与系统昵称映射，统一按“岗位命名与昵称映射”小节执行。
- 每个岗位必须有明确职责、输入材料、输出交付物和禁止越界范围。
- 不为了“看起来协作”而创建 Agent；只有当某个岗位能产生明确增量价值时，才启用对应子 Agent。
- 启用多 Agent 协作模式时，至少创建 3 个及以上子 Agent；如果任务不足以支撑 3 个有明确职责的岗位，应保持 Codex 单 Agent 执行，不启用多 Agent 模式。
- 子 Agent 数量应根据工作需要和任务进度动态调整：早期可偏向产品、研究、设计和架构；中期可增加前端、后端、数据、测试等执行岗位；后期可增加 Review、文档、发布、复盘等收口岗位。
- 一旦用户确认启用多 Agent，不要刻意压缩必要的独立复核岗位；当某个工作方向需要独立判断、独立交付或独立审查时，应创建对应岗位，而不是让一个子 Agent 兼任过多职责。
- 同一个问题只分配给一个主责岗位，其他岗位最多做补充审查，避免多人重复劳动。
- Codex 必须像项目经理一样拆解任务、分配工作、控制范围、追踪结果、处理冲突、整合交付。

### 需求清单先转启动提示词

如果用户给出的是需求清单、功能清单、想法列表或零散目标，Codex 不得直接进入实现或派发任务。必须先将其整理并优化为适合启动项目的结构化提示词或项目 Brief，再继续下一步。

该启动提示词或项目 Brief 应尽量包含：

- 项目目标。
- 适用场景和目标用户。
- 功能范围和非功能范围。
- 优先级和 MVP 边界。
- 关键约束、风险和假设。
- 多 Agent 协作目标。
- 推荐岗位和分工方式。
- 阶段划分和交付物。
- 验收标准。

如果需求清单存在明显缺口，Codex 应先提出少量关键澄清问题；如果缺口不影响启动，则应明确假设并继续。

### 启动编组说明

启用该模式时，Codex 应优先形成一个简短的协作编组说明，包括：

- 本次为什么需要多 Agent。
- 本次启用哪些岗位，以及为什么这些岗位数量合适。
- 如果工具显示随机昵称，列出“系统昵称 -> 岗位名称”的映射表。
- 当前阶段是否需要增减子 Agent。
- 如果线程、额度或子 Agent 失败，PM 如何降级接管。
- 每个岗位负责什么。
- 每个岗位不负责什么。
- 各岗位输出如何被 Codex 使用。
- 如何避免重复工作和 token 浪费。

如果任务不需要至少 3 个职责清晰、彼此互补的岗位，则不要启用多 Agent 协作模式。启用后，Codex 应随着任务推进持续判断是否需要新增、关闭或调整子 Agent。

### 岗位命名与昵称映射

每一个被创建或调用的子 Agent 都必须依据其真实职责命名为有意义的岗位名称，例如架构师、产品经理、UI/UX 设计师、前端工程师、后端工程师、测试工程师、安全审查员、文档工程师、研究员等；不得使用“Agent 1”“助手 A”“子任务处理者”等无意义名称。

如果底层工具或界面自动分配了无法控制的随机昵称，Codex 必须立即建立“系统昵称 -> 岗位名称”的映射，并在所有协作说明、任务派发、进度更新、日志和最终汇总中优先使用岗位名称。

如果系统昵称无法改名，Codex 必须在内部记录和用户可见输出中使用“岗位名称（系统昵称）”格式，避免用户看到一组无业务意义的名字后无法判断职责。

### Codex 主控职责

启用多 Agent 协作后，Codex 仍然是唯一主控，必须负责：

- 阅读并遵守 `{{WORKSPACE_ROOT}}\01_全局工作台.md` 中的项目工作流、复利与踩坑日志、项目创建规则。
- 判断当前任务是否需要创建新的项目文件夹。
- 创建和维护项目 Brief、任务拆分、执行日志和复盘。
- 当用户提供需求清单时，按“需求清单先转启动提示词”小节执行，再进入编组、派发或实现。
- 将复杂任务拆分为边界清楚、输入明确、输出可验收的子任务。
- 决定哪些子任务适合交给其他 Agent。
- 为其他 Agent 编写任务说明。
- 审查其他 Agent 的输出、代码改动、文档和测试结果。
- 处理 Agent 之间的冲突和结果整合。
- 运行或确认必要验证。
- 向用户做最终交付说明。
- 设计并执行多 Agent 失败时的降级路径，避免项目被单个岗位阻塞。

Codex 不得因为其他 Agent 声称完成，就跳过审查、测试或最终判断。

### 子 Agent 的岗位角色

子 Agent 必须以具体岗位身份参与任务，而不是以笼统的“另一个助手”身份参与。子 Agent 的岗位名称必须直接反映其职责和交付物；随机昵称处理统一按“岗位命名与昵称映射”小节执行。

可按任务需要设置以下岗位。岗位应尽量细分，避免一个子 Agent 同时承担多个性质不同的职责。

代码开发与工程岗位：

- 产品经理：澄清需求、用户场景、验收口径和优先级。
- 技术负责人：统筹技术路线、关键取舍、跨模块协调和技术债控制。
- 架构师：负责系统边界、技术方案、模块拆分、风险取舍。
- 前端工程师：负责限定范围内的页面、组件、状态和浏览器验证。
- 后端工程师：负责限定范围内的接口、服务逻辑、数据模型和集成验证。
- 数据库工程师：负责数据模型、索引、迁移、查询性能和数据一致性。
- DevOps 工程师：负责本地运行、构建、部署、环境变量、CI/CD 和运行脚本。
- 测试工程师：负责测试计划、用例补充、回归验证、失败定位。
- 安全审查员：负责权限、凭据、输入校验、数据暴露和高风险操作审查。
- 性能工程师：负责性能瓶颈、加载速度、缓存策略、并发和资源消耗。
- Review 工程师：负责代码审查、边界检查、潜在回归和遗漏风险。

设计与创意岗位：

- UI/UX 设计师：负责信息架构、界面布局、交互体验、视觉一致性。
- 视觉设计师：负责视觉风格、色彩、字体、图形语言、品牌一致性。
- 品牌策略师：负责品牌定位、语气、差异化表达和传播一致性。
- 内容策划：负责页面文案、信息层级、营销表达、用户引导。
- 创意总监：负责创意方向、概念统一、表达质量和审美把关。
- 原型设计师：负责页面流程、低保真/高保真原型、关键交互说明。

日常工作与知识生产岗位：

- 研究员：负责资料收集、竞品分析、方案比较、证据整理。
- 数据分析师：负责指标拆解、表格分析、趋势判断、可视化建议。
- 文档工程师：负责 README、SOP、交付说明、复盘材料。
- 项目助理：负责会议纪要、任务清单、进度同步、待办追踪。
- 质量审校员：负责错别字、格式、术语统一、交付材料一致性。
- 法务/合规审查员：负责合规风险、版权风险、隐私条款和敏感表述提醒。
- 财务/商业分析师：负责成本估算、收益模型、预算拆分和商业可行性提示。

每次启用时不必使用全部岗位，但应根据任务阶段和真实需要灵活选择、增减和关闭子 Agent。

每个子 Agent 必须有唯一主责，并使用真实有意义的岗位名称。启用多 Agent 协作时，子 Agent 数量不得少于 3 个；若无法合理设置至少 3 个岗位，Codex 应回退为单 Agent 执行。随着任务推进，Codex 应动态增加、调整或关闭子 Agent，避免职责过粗、岗位空转或重复劳动。子 Agent 不得擅自扩大任务范围，不得越权修改未授权文件，不得跳过 Codex 直接向用户交付最终结论。

### 派发任务标准

Codex 给其他 Agent 派发任务时，必须具体、有限、可验收，不得派发模糊任务。任务说明必须包含：

- 项目路径。
- 任务目标。
- 背景说明。
- 允许修改的文件或目录。
- 禁止修改的文件或目录。
- 明确禁止做的事情。
- 验收标准。
- 建议运行的验证命令。
- 完成后必须回传的内容。

禁止派发这类模糊任务：

- “帮我优化整个项目”
- “你看着改”
- “全面重构”
- “把所有问题都修了”
- “随便整理一下”

这类任务必须先由 Codex 拆解成具体、有限、可验收的子任务，再按上述字段派发。

### 资源限制与降级路径

多 Agent 协作依赖线程、额度、模型可用性和工具状态，不能假设所有岗位一定能完整返回。

启用多 Agent 协作时，Codex 必须预先设计降级路径：

- 不要一次性无边界启动全部岗位；优先按阶段、优先级和依赖关系分批创建子 Agent。
- 子 Agent 完成任务后，应及时关闭或释放，避免占用线程名额影响后续关键岗位。
- 如果某个子 Agent 因线程上限、额度限制、工具错误或超时失败，Codex PM 必须记录该岗位缺口和风险。
- 对非核心岗位失败，Codex PM 可以继续推进，并在最终交付中说明缺口、影响和未验证风险。
- 对核心岗位失败，Codex PM 应优先接管关键判断、验收或实现；必要时重新创建职责更小的替代子 Agent。
- 不得因为等待单个失败岗位而让整个项目停滞；PM 必须判断该岗位是否阻塞主线。
- 重新派发失败岗位前，应缩小任务范围，避免重复触发线程或额度问题。
- 所有失败、接管、跳过、重派发和风险接受，都应记录到 `06_Logs/agent-review.md` 或对应项目日志。

未来复用规则：多 Agent 任务必须设计“岗位失败时 PM 如何接管”的降级路径。

### 并行修改限制

默认不允许多个 Agent 同时修改同一批核心文件。

如果确实需要并行处理，必须满足：

- 每个 Agent 的修改范围相互独立。
- 每个 Agent 有明确的文件边界。
- 不同 Agent 不同时修改同一文件，除非用户明确批准。
- Codex 最终统一 review、合并和验证。
- 出现冲突时，由 Codex 判断处理方案，不得自动覆盖任何一方改动。

如果 Codex 工作过程中发现用户、其他独立 Agent、外部工具或编辑器同时修改了文件：

- 先区分哪些改动是 Codex 自己产生的，哪些是外部改动。
- 不得覆盖、回滚或删除外部改动，除非用户明确要求。
- 如果外部改动与当前任务不冲突，应基于最新文件继续工作。
- 如果外部改动影响当前任务，应先阅读并适配；无法安全合并时，停下来向用户说明冲突和可选处理方案。
- 最终回复中应说明已观察到的外部并发修改及其处理方式。

### 回传内容要求

其他 Agent 完成任务后，必须回传：

- 完成了什么。
- 修改了哪些文件。
- 为什么这样修改。
- 是否超出原任务范围。
- 运行了哪些验证。
- 哪些验证没有运行，以及原因。
- 当前仍存在的风险、疑问或待确认事项。
- 需要 Codex 继续整合或判断的事项。

### Codex 验收规则

Codex 收到其他 Agent 的结果后，必须检查：

- 是否越界修改文件。
- 是否引入无关重构。
- 是否修改依赖、配置、数据库、CI、权限、部署等高风险内容。
- 是否删除、弱化、跳过或伪造测试。
- 是否符合项目已有风格和架构。
- 是否需要补充测试、文档、日志或复盘。
- 是否与用户原始目标一致。
- 是否存在未说明的风险。

只有 Codex 完成最终审查和验证后，才能向用户交付结果。

### 多 Agent 协作日志

启用多 Agent 协作时，项目文件夹中应创建或维护以下日志：

```text
06_Logs/
  agent-handoffs.md
  agent-decisions.md
  agent-review.md
```

其中：

- `agent-handoffs.md` 记录每次派发给其他 Agent 的任务说明。
- `agent-decisions.md` 记录关键方案选择、取舍和原因。
- `agent-review.md` 记录 Codex 对其他 Agent 输出的审查结果，包括岗位失败、PM 接管、风险接受和降级处理。

如果过程中出现踩坑、误判、返工、冲突或可复用经验，应同步记录到：

```text
{{WORKSPACE_ROOT}}\01_全局工作台.md
```

### 高风险禁区

未经用户明确授权，任何 Agent 都不得：

- 删除文件或目录。
- 重置 Git 历史。
- 强制覆盖用户改动。
- 修改全局配置。
- 升级依赖或切换技术栈。
- 操作生产环境。
- 使用真实凭据、密钥或线上数据。
- 发布、部署、推送镜像、创建正式 release。
- 修改其他 Agent 或用户正在处理的文件。

### 最终交付要求

多 Agent 协作任务的最终回复必须由 Codex 给出，并说明：

- 本次是否启用了多 Agent 协作。
- 每个 Agent 分别负责什么。
- Codex 最终整合了哪些内容。
- 实际运行了哪些验证。
- 哪些验证没有运行，以及原因。
- 仍然存在的风险或后续建议。

Codex 必须对最终结果负责，不得把责任转移给其他 Agent。
