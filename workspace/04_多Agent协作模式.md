# 多 Agent 协作模式

启用多 Agent 后必须遵守本文件。启动条件见 `{{CODEX_AGENTS}}`。

## 定位与启用

多 Agent 是显式高级模式，用于认知分工、方案竞争和独立复核，不是普通工程执行或省额度工具。复杂项目不会自动启用。

用户明确要求多 Agent 协作时，视为已经授权，Codex 读取本文件后直接编组，不重复请求确认。

Codex 主动建议启用时，必须先说明：

- 启用理由、岗位及各自交付物。
- 单 Agent 或 CC-Switch worker 为什么不够。
- 线程与额度成本、关键风险和降级方案。
- 将读取并遵守本文件。

由 Codex 主动建议时，用户确认后才能编组。任务至少要能拆出一个可独立推进、边界清楚且有增量价值的子任务；纯顺序工作、必须共享同一写入面的工作，或 Codex 自己更快完成的任务不派发。

子 Agent 通常为 1 至 3 个，以当前工具返回的槽位为准，并始终为 Codex 主控保留一个槽位。岗位越多不等于越快；新增岗位必须有会被主线使用的独立交付物。CC-Switch worker 是受限执行工具，不计入子 Agent 数量。

## 主控关系

- Codex 是唯一 PM、决策者和最终验收者。
- 子 Agent 提供专家判断、阶段成果或独立审查。
- CC-Switch worker 只执行边界明确、可回滚、可验证的小块工作。
- 任何 Agent 或 worker 的输出都不能绕过 Codex 成为最终交付。
- 高风险判断、冲突处理、结果整合和最终验证由 Codex 负责。

项目创建、目录、日志与复盘遵守 `{{WORKSPACE_ROOT}}\01_全局工作台.md`；worker 调用遵守 `{{WORKSPACE_ROOT}}\05_CC-Switch-Worker-MCP规则.md`。

## 启动准备

任务符合 `{{WORKSPACE_ROOT}}\01_全局工作台.md` 的项目条件时，先整理项目 Brief；一次性调研、评审或轻量协作只需在当前任务中写清启动基线。用户提供需求清单、功能列表或零散目标时，启动基线至少包含：

- 目标用户、场景和项目目标。
- 功能范围、非功能范围和 MVP 边界。
- 约束、风险、假设和验收标准。
- 阶段、交付物、岗位及协作关系。

不影响启动的缺口可明确假设；会改变目标或范围的缺口先提问。

编组说明应简短列出：启用原因、岗位、职责边界、交付物、依赖关系、失败时的 PM 接管方式，以及如何避免重复工作。

## 调度闭环

每一批协作按以下顺序执行：

1. Codex 先确定主线、依赖关系和完成标准。
2. 只派发能独立推进的子任务；存在前置依赖时分批启动，不把所有槽位一次占满。
3. 只读调研和独立复核可以并行。写入任务必须有互不重叠的文件边界，同一文件同时只有一个写入者。
4. 子 Agent 运行时，Codex 继续处理整合、验证或其他不冲突的工作，不原地等待。
5. 结果返回后，Codex 检查结论、证据、范围和验证，再决定接纳、纠偏、补派或接管。
6. 当前批次收口后再启动下一批，为关键复核和失败重派保留容量。

能复用的上下文和检查结果只读取一次。只有独立产出能够抵消额外上下文、等待和整合成本时，才增加岗位。

## 岗位规则

- 岗位按真实职责命名，如产品经理、架构师、前端工程师、测试工程师、安全审查员、研究员或文档工程师；不得使用“Agent 1”等无意义名称。
- 工具自动分配随机昵称时，立即建立“系统昵称 -> 岗位名称”映射；用户可见内容使用“岗位名称（系统昵称）”。
- 每个岗位只有一个主责，职责、输入、输出和禁区必须明确。
- 同一问题只设一个主责岗位，其他岗位最多补充审查。
- 岗位随阶段增减；完成后及时释放，避免空转和占用线程。
- 子 Agent 不得扩大范围、修改未授权文件或直接向用户交付最终结论。

常用岗位按需选择：

- 产品与研究：产品经理、研究员、数据分析师、商业或合规审查员。
- 技术与实现：技术负责人、架构师、前端、后端、数据库、DevOps、性能工程师。
- 质量与交付：测试工程师、安全审查员、Review 工程师、文档工程师、质量审校员。
- 设计与内容：UI/UX、视觉、品牌、内容策划、原型设计师、创意总监。

不要为了凑岗位而启用无增量价值的角色。

## 派发标准

每个子任务必须具体、有限、可验收，任务包只保留完成工作所需的信息：

- 岗位、唯一目标和必要背景。
- 项目路径，以及应该读取的文件或上下文范围。
- 允许与禁止修改的文件或目录。
- 交付物、验收标准、验证命令和停止条件。
- 回传格式：结论、证据、改动、验证和风险。

“优化整个项目”“你看着改”“全面重构”“修复所有问题”等任务必须先拆解。

背景与当前对话紧密耦合时传递完整上下文；边界清楚的任务传递最近对话或不继承对话，但任务包必须自足。不同 Agent 不做相同调研，不重复读取同一批大文件；派发前确认输出用途。

## 并行与写入所有权

- 只读任务可以并行，但不得用相同输入重复回答同一问题。
- 写入任务必须有互不重叠的文件边界；同一文件在交接完成前只有一个写入者。
- Codex 统一 review、整合和验证；冲突时不得自动覆盖任一方。
- 发现用户、外部工具或其他 Agent 的并发改动时，先区分来源。无冲突则基于最新文件继续；有冲突则阅读并适配，无法安全合并时告知用户。
- 未经明确要求，不覆盖、回滚或删除外部改动。最终回复说明观察到的并发修改及处理方式。

## 运行中控制

- 方向正确时让子 Agent 自主完成，不持续轮询；Codex 同时推进主线工作。
- 运行中的任务需要纠偏时使用 `send_message`；任务已空闲且需要继续处理时使用 `followup_task`。
- 只有继续运行会越界、产生错误改动或明显浪费资源时，才使用 `interrupt_agent`。
- 仅在主控没有其他可推进工作时使用 `wait_agent`，单次等待不超过 60 秒，并保持必要的用户进度更新。
- 子 Agent 默认不再自行编组。确有独立子任务、仍有可用槽位且任务包已授权时，才允许向下委派。

## 回传与验收

子 Agent 回传：

- 完成内容、修改文件和原因。
- 是否越界。
- 已运行与未运行的验证。
- 剩余风险、疑问和需要 Codex 判断的事项。

Codex 必须检查：

- 文件范围和用户目标是否一致。
- 是否有无关重构、依赖或配置变化。
- 是否触及数据库、CI、权限、部署等高风险表面。
- 是否删除、弱化、跳过或伪造测试。
- 是否符合项目架构与风格，是否缺测试、文档或日志。

只有 Codex 审查并完成最终验证后，任务才算完成。

## 失败与降级

- 按阶段和依赖分批创建 Agent，不一次性无边界启动全部岗位。
- Agent 因线程、额度、工具或超时失败时，记录岗位缺口、影响和风险。
- 非核心岗位失败可继续推进，并在交付中说明缺口。
- 核心岗位失败由 Codex 接管；必要时把任务缩小后重新派发。
- 不让单个失败岗位阻塞整条主线。
- 正式项目把失败、接管、跳过、重派和风险接受写入 `06_Logs/agent-review.md`；一次性任务在最终回复中说明。

## 协作日志

没有正式项目目录的一次性任务不创建协作日志，进度更新和最终回复即为记录。

正式项目默认只维护：

```text
06_Logs/
  agent-review.md  # 派发、关键决策、审查、失败与降级
```

只有协作持续多个阶段、单文件已经难以检索时，才拆出 `agent-handoffs.md` 和 `agent-decisions.md`。日志保留结论和证据，不保存完整对话转录。

通用经验按 `{{WORKSPACE_ROOT}}\01_全局工作台.md` 的知识库与复盘规则处理，不堆回本文件。

## 高风险禁区

未经用户明确授权，任何子 Agent 都不得：

- 删除文件、重置 Git、覆盖用户改动。
- 修改全局规范、账号配置或关键安全策略。
- 升级依赖、切换技术栈或操作生产环境。
- 使用真实凭据、密钥或生产数据。
- 发布、部署、推送镜像或创建正式 release。
- 修改用户或其他 Agent 正在处理的文件。

## 最终交付

最终回复由 Codex 给出，说明启用的岗位及分工、整合内容、实际验证、未运行的检查、剩余风险和降级情况。Codex 对最终结果负责。
